\section{Introduction}
Mutation testing measures whether a test suite can detect seeded faults rather than merely execute the corresponding lines. For Milestone 5, our goal is to use PIT on Gson to assess the quality of tests around a coherent parsing and streaming subsystem, document the setup process, and establish a reproducible baseline before deeper surviving-mutant analysis in later experiments.

This draft focuses on three questions. First, can PIT be configured reproducibly on Gson with a justified and non-trivial target scope? Second, once that scope is fixed, what does the first full mutation baseline reveal about the current test suite? Third, can targeted test additions kill selected surviving mutants and measurably improve the baseline score? The emphasis in this revision is therefore on configuration, scope calibration, accurate baseline reporting, and focused surviving-mutant analysis.

\section{Background}
Mutation testing works by generating small source-level or bytecode-level changes, running the existing tests, and classifying each mutant as killed, survived, not covered, or timed out. Unlike line coverage, mutation results say something about the strength of the assertions and the sensitivity of the tests to behavior changes. A project can execute nearly all lines in a subsystem and still allow important mutants to survive.

PIT is a practical Java mutation testing tool because it performs coverage analysis first and only runs tests relevant to each mutant. It also produces HTML and XML reports that expose both high-level project metrics and per-mutant details. This matters for Gson because the selected scope contains both deeply stateful parsing code and smaller adapter or bridge classes. Those classes are all part of one feature area, but they differ substantially in how much mutatable logic they contain.

Equivalent mutants remain a known limitation. A mutant can survive either because tests are weak or because the mutated program is behaviorally indistinguishable from the original. Distinguishing those cases is part of later milestone work, but even in Experiments 1--3 it is important to avoid over-interpreting raw survival counts.

\subsection{Scope Selection, Justification, and Hypothesis}
The selected scope expands outward from Gson's streaming subsystem rather than redirecting the analysis to a different part of the project. The core package \texttt{com.google.gson.stream} contains the most stateful reader and writer logic in the chosen slice, especially \texttt{JsonReader} and \texttt{JsonWriter}. We then added adjacent parser and tree-bridge classes that sit directly on top of that streaming layer: \texttt{JsonParser}, \texttt{JsonStreamParser}, \texttt{Streams}, \texttt{JsonTreeReader}, \texttt{JsonTreeWriter}, and \texttt{JsonElementTypeAdapter}.

This is a feature-based selection. The classes together implement one coherent concern: parsing JSON text or trees, traversing it, and serializing it back out. The scope is also complexity-oriented. The reader/writer classes are state machines with many branching conditions, while the tree bridge classes expose smaller but still meaningful control-flow decisions around adapter behavior, null handling, and reader/writer transitions.

Before running the full baseline, our hypothesis was that this subsystem would show very high line coverage and a moderately high mutation score because Gson already ships dedicated reader, writer, parser, and tree-adapter tests. We nevertheless expected some surviving mutants in control-flow edges, leniency handling, and return-value behavior, especially in the bridge classes where tests are often more scenario-driven than assertion-heavy.

\subsection{Pre-run Validation and Setup Challenges}
We first validated that the selected module had a green test suite before any mutation run. The Docker-based verification run completed successfully with 4,586 tests executed, 0 failures, 0 errors, and 19 skipped tests in 38.081 seconds. This satisfied the milestone requirement to confirm a passing suite before PIT.

To verify the PIT configuration itself, we then performed a smoke run on the original streaming slice, namely \texttt{com.google.gson.stream.*} together with \texttt{com.google.gson.JsonStreamParser}. This was intentionally used as a calibration step because it is the narrowest coherent subset of the final scope and exercises the same parser/streaming behavior. The smoke run produced a valid HTML report, generated 687 mutants, and completed in 55 seconds of PIT time (1 minute 10 seconds wall-clock through Maven), which is in the assignment's marginal range and therefore suggested that the scope could be widened further.

Two practical setup issues mattered. First, Gson must be executed from the inner module directory \texttt{gson/gson}; running PIT from the repository root would not target the actual library module under test. Second, the initial streaming-only selection contained several files that were in scope but effectively non-mutatable under PIT defaults, including constant-only or metadata-only files. Because that reduced the effective mutatable surface and kept runtime low, we expanded the scope to adjacent parser/tree classes while preserving the same subsystem focus. This calibration step moved the final baseline runtime into the milestone's preferred 2--30 minute band.

\paragraph{Original vs. mutated code.}
The original code is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
peekedString = nextQuotedValue(p == PEEKED_SINGLE_QUOTED ? '\'' : '"');
\end{lstlisting}

The mutant replaces the equality check \texttt{p == PEEKED\_SINGLE\_QUOTED} with \texttt{false}, so the ternary always selects \texttt{'"'} and ignores single-quoted input.

\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
peekedString = nextQuotedValue(false ? '\'' : '"');
// equivalent behavior observed by PIT mutant:
peekedString = nextQuotedValue('"');
\end{lstlisting}

\paragraph{M1. Survival explanation.}
The existing test suite does not call \texttt{nextInt()} on a single-quoted string value. The tests around \texttt{JsonReaderTest.java} exercise unquoted numeric literals and double-quoted numeric strings, but none of them set \texttt{Strictness.LENIENT} and pass an input like \texttt{['123']}. Because PIT's mutant changes the conditional on line 1334 specifically, the path that requires the single-quoted delimiter \texttt{'\''} is never exercised.

\paragraph{M2. Semantic impact assessment.}
A concrete runtime scenario is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
JsonReader reader = new JsonReader(new StringReader("['123']"));
reader.setStrictness(Strictness.LENIENT);
reader.beginArray();
int value = reader.nextInt();
\end{lstlisting}

With the original code, \texttt{nextQuotedValue('\\'')} reads the characters between the single quotes and returns \texttt{123}. With the mutant, \texttt{nextQuotedValue('"')} searches for a closing double quote instead of a single quote, so parsing fails or produces the wrong value. The observable behavior changes from returning \texttt{123} to throwing a parse/number-format failure.

\paragraph{M3. Killability verdict.}
This mutant is \textbf{killable -- missing test}. No test reaches the single-quoted lenient path. The added test is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
@Test
public void testNextIntFromSingleQuotedString() throws IOException {
  JsonReader reader = new JsonReader(reader("['123']"));
  reader.setStrictness(Strictness.LENIENT);
  reader.beginArray();
  assertThat(reader.nextInt()).isEqualTo(123);
  reader.endArray();
}
\end{lstlisting}

This test is sufficient because it asserts the exact return value from the previously untested single-quoted branch. PIT confirmation from the follow-up run reports line 1334 as killed by \texttt{testNextIntFromSingleQuotedString}.

\paragraph{Original vs. mutated code.}
The original code is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
@Override
public void close() throws IOException {
  peeked = PEEKED_NONE;
  stack[0] = JsonScope.CLOSED;

Mutant code: The call to in.close() is completely removed. The JsonReader sets its own state to CLOSED but never delegates close() to the underlying Reader.

\paragraph{M1. Survival explanation.}
The only covering test in the suite is \texttt{testPrematurelyClosed()}, which verifies only that the \texttt{JsonReader} state changes to closed and that subsequent operations throw \texttt{IllegalStateException}. It does not observe whether the wrapped \texttt{Reader.close()} method is actually invoked, so the mutant survives all seven covering tests.

\paragraph{M2. Semantic impact assessment.}
A concrete runtime scenario is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
FileInputStream fis = new FileInputStream("data.json");
InputStreamReader isr = new InputStreamReader(fis);
JsonReader reader = new JsonReader(isr);
reader.beginObject();
reader.close();
\end{lstlisting}

With the original code, closing the \texttt{JsonReader} propagates to the underlying stream and releases the file descriptor. With the mutant, the underlying stream stays open. The difference is observable if the caller checks that the wrapped reader was closed or attempts to use the wrapped stream after closing.

\paragraph{M3. Killability verdict.}
This mutant is \textbf{killable -- weak assertion}. The existing test covers the method but not the side effect on the wrapped reader. The added test is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
@Test
public void testCloseClosesUnderlyingReader() throws IOException {
  final boolean[] closeCalled = {false};
  Reader inner =
    new StringReader("[1, 2, 3]") {
      @Override
      public void close() {
      closeCalled[0] = true;
      super.close();
      }
    };
  JsonReader reader = new JsonReader(inner);
  reader.beginArray();
  reader.close();
  assertThat(closeCalled[0]).isTrue();
}
\end{lstlisting}

This assertion is sufficient because it directly checks the delegated close behavior that the mutant removes. PIT confirmation from the follow-up run reports line 1372 as killed by \texttt{testCloseClosesUnderlyingReader}.

\paragraph{Original vs. mutated code.}
The relevant original code is:
\begin{lstlisting}[basicstyle=\ttfamily\small,breaklines=true]
while ((total = in.read(buffer, limit, buffer.length - limit)) != -1) {
  limit += total;

  if (lineNumber == 0 && lineStart == 0 && limit > 0 && buffer[0] == '\ufeff') {
    pos++;
    lineStart++;
    minimum++;
  }

  if (limit >= minimum) {
    return true;
  }

\end{lstlisting}

The mutant changes the BOM guard from \texttt{limit > 0} to \texttt{limit >= 0}.

\paragraph{M1. Survival explanation.}
The mutated condition sits inside a loop that only executes after \texttt{Reader.read(...)} returns a value other than \texttt{-1}. A successful read returns at least one character when the length argument is positive, and the buffer length available in this method is positive when the loop executes. Therefore the code reaches line 1504 only with \texttt{limit >= 1}. Because the existing suite does not introduce an invalid reader that violates the Java contract, the surviving tests all observe the same behavior.

\paragraph{M2. Semantic impact assessment.}
No observable difference exists under any contract-compliant input. The program state guarantees that \texttt{limit} is already positive when the boundary check runs, so \texttt{limit > 0} and \texttt{limit >= 0} are equivalent at the mutation site. A counter-example using a custom reader that returns 0 characters was considered, but the Java \texttt{Reader.read(char[], int, int)} contract does not permit that return value when the length argument is positive.

This mutant is \textbf{likely equivalent}. The mutation changes the syntax of the comparison but not the reachable semantics of the method. The evidence table is:

\subsubsection{Cost-Benefit: All Operators vs.\ Selective Subset}

Running all default operators generated 945 mutants and took about 4 minutes and 9 seconds. That is a reasonable runtime for this scope. That said, just 5 operators produced over 83\% of all mutants, and the operators that hit 100\% kill rates did not reveal anything new to fix.

The most useful operators were the weakest ones. \texttt{ConditionalsBoundaryMutator} and \texttt{NullReturnValsMutator} pointed to real gaps that are still worth addressing, as seen in the modest score improvement from 85.6\% to 86.1\% after Experiment 3.

Selective mutation testing makes more sense when the project is large and runtimes exceed 30 minutes, when the goal is finding and fixing test gaps rather than just measuring coverage, or when re-running PIT after changes where some operators are already at 100\%. For this project the runtime was short enough that running everything made sense, but on a larger codebase focusing on the weakest operators would save time while still finding the most important issues.

\section{Discussion}
The full baseline largely confirmed the pre-run hypothesis. The selected Gson subsystem has very high line coverage and a strong overall mutation score, which indicates that the existing test suite is not merely executing code but is detecting many seeded faults. That is especially clear in the streaming layer, where \texttt{JsonReader} and \texttt{JsonWriter} account for most mutants and are still tested strongly.

At the same time, the calibration step mattered. The original streaming-only selection was coherent but too narrow in effective mutatable surface because several selected files were trivial under PIT's default mutators. Expanding to adjacent parser and tree-bridge classes preserved the architectural focus while improving experimental value. The final 4:09 runtime is much more appropriate for the milestone than the 55-second smoke run.

The results also show why mutation testing gives a different picture from coverage alone. Even with 96\% mutated-class line coverage, 132 mutants survived in the Experiment 2 baseline. Experiment 3 then demonstrated that some of these survivors were killable with targeted tests that strengthened specific behavioral assertions. At the same time, the likely-equivalent \texttt{fillBuffer} mutant illustrates that not all surviving mutants indicate weak tests.

Together, Experiments 2 and 3 provide a clearer quality signal: the suite is already strong at subsystem level, still has actionable assertion gaps in edge-case semantics, and includes a smaller set of survivors that are better interpreted as likely equivalent rather than missing-test defects.

\section{Lessons Learned}
Four lessons are clear from Experiments 1--3. First, scope selection must be guided by mutatable logic rather than raw source lines alone. Second, a subsystem-focused scope can stay small enough to run repeatedly while still generating nearly a thousand mutants if the selected classes actually contain branching behavior. Third, high coverage by itself can overstate adequacy; surviving mutants expose behavior not yet strongly asserted. Fourth, surviving-mutant triage is essential: targeted tests can eliminate real weaknesses, while explicit equivalent-mutant analysis avoids misclassifying unreachable semantic differences as test defects.

\section{Conclusion}
For this selected Gson subsystem, PIT setup was reproducible and produced a useful mutation-testing workflow across three experiments. The Experiment 2 baseline yielded 945 mutants with an 85.6\% mutation score by the milestone formula, 86\% PIT test strength, 96\% line coverage, and a 4:09 wall-clock runtime. Experiment 3 then added two targeted tests that killed selected surviving mutants and improved the mutation score to 86.1\% (804 killed, 130 survived, out of 957 generated mutants in the rerun).

These results show that mutation testing is practical and informative for a production Java project when the scope is chosen as a coherent feature slice with meaningful logic and a runtime in the low-minute range. The next report revisions will build on this foundation with Experiment 4 and Experiment 5 analyses using the same repository and target scope.
